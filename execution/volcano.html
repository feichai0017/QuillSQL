QuillSQL Internals

Updated Volcano Execution Model

Once the Query Planner has produced an optimized vectorized tree, the Execution Engine consumes that metadata and produces results. We keep the traditional Volcano terminology for plan shapes, but the actual runtime now relies on chunk-oriented operators instead of the legacy tuple-at-a-time iterator.

1. VecPlan metadata

VecPlan is now the planner’s output: each variant stores the expressions, schemas, table/index references, join properties, etc. that define a particular operator. Unlike the original implementation, these structs are now pure data containers. They do not own scanners, mutexes, or iterators; all runtime state lives in the chunk operators that get built on top of them. This keeps the planner/optimizer API stable while allowing the executor to evolve independently.

2. Chunk-based Operator trait

Execution now happens through the chunk Operator interface in src/execution/operator.rs:

#![allow(unused)]
fn main() {
pub trait Operator {
    fn create_state(&self) -> Box<dyn OperatorState>;
    fn open(&self, ctx: &mut ExecutionContext, state: &mut dyn OperatorState) -> Result<()>;
    fn get_chunk(
        &self,
        ctx: &mut ExecutionContext,
        state: &mut dyn OperatorState,
        chunk: &mut DataChunk,
    ) -> Result<ChunkStatus>;
    fn rescan(&self, ctx: &mut ExecutionContext, state: &mut dyn OperatorState) -> Result<()>;
    fn close(&self, ctx: &mut ExecutionContext, state: &mut dyn OperatorState);
}
}
  • OperatorState stores per-execution mutable state (cursors, hash tables, pending tuples). It is opaque to the engine and downcast by the operator.
  • DataChunk (default capacity: 1024 rows) is the batch abstraction exchanged between operators. It is currently row-based but designed for vectorization.
  • ChunkStatus::{MoreData, Finished} tells callers whether more chunks may follow.
  • ExecPlan::from_vec rewrites every node (SeqScan, Filter, Project, Limit, DDL, etc.) into concrete chunk operators. Because the Vec structs are metadata-only, no legacy iterator code remains to maintain.

ExecutionEngine::stream_exec_plan opens the root operator, hands it a state object, and returns a ChunkStream iterator. ExecutionEngine::execute simply drains this stream into a Vec<Tuple> for callers that still expect eagerly materialized results.

3. ExecutionContext Helpers

Both legacy and chunk operators receive a mutable ExecutionContext, which exposes:

  • Catalog lookups.
  • TransactionManager / TxnRuntime for lock acquisition, MVCC snapshots, and undo logging.
  • StorageEngine handles (TableAccess, IndexAccess, TupleStream) via table_stream, index_stream, etc.
  • Expression helpers (eval_expr, eval_predicate) that return ScalarValues.
  • DML helpers (prepare_row_for_write, apply_update/delete, insert_tuple_with_indexes).

Operators stay small because they delegate MVCC and storage concerns to the context.

4. Operator Anatomy (Legacy vs Chunk)

SeqScan / IndexScan

  • Legacy: init opens a TableIterator; next pulls (rid, meta, tuple) one at a time, performs visibility checks, and returns visible tuples.
  • Chunk: SeqScanOperator (and IndexScanOperator for secondary indexes) keeps an explicit prefetch buffer, batches MVCC visibility checks, and acquires whatever locks are required once per batch instead of once per tuple.

Filter

  • Legacy: Loops calling child.next(), evaluates a predicate, returns the tuple when the predicate is true.
  • Chunk: FilterOperator consumes child chunks, evaluates predicates row by row, and either places tuples into the current chunk or buffers them if the chunk is full.

Project

  • Legacy: Evaluates projection expressions on each tuple returned by the child.
  • Chunk: ProjectOperator evaluates expressions across the current chunk, producing a new chunk with projected tuples.

Nested Loop Join

  • The new chunk operator materializes the right input once (reusing existing chunk abstractions), then streams left-side chunks. For each left tuple it iterates the cached right tuples, merges them, and evaluates the join predicate before appending to the output chunk.
  • Because the right side is buffered, rescan becomes cheap and downstream operators can keep chunk-level scheduling, while the overall behaviour still matches the legacy Volcano execution order.

DDL / Analyze

  • CreateTable, CreateIndex, DropTable, DropIndex, and Analyze run as lightweight chunk operators. Each operator acquires the relevant locks via ExecutionContext::txn(), runs exactly once, and immediately reports ChunkStatus::Finished, allowing DDL to coexist with streaming pipelines.

5. Example: SELECT name FROM users WHERE age > 30

Vec plan: Project -> Filter -> SeqScan.

  1. ExecutionEngine::stream builds an ExecPlan, wraps the SeqScan/Filter/Project nodes in chunk operators, and opens the root ProjectOperator.
  2. ProjectOperator::get_chunk requests a chunk from its child (FilterOperator).
  3. FilterOperator::get_chunk requests a chunk from SeqScanOperator.
  4. SeqScanOperator fetches raw rows from the heap, filters by MVCC visibility, and fills the chunk.
  5. The chunk flows back up: Filter evaluates age > 30, Project keeps only name.
  6. The engine yields the resulting DataChunk to the caller. ChunkStream keeps pulling until operators report Finished.

The control flow is identical to the tuple-at-a-time model, except we operate on batches and the engine can stream results without buffering the entire query.

For Study & Discussion

  1. Push vs Pull – What would it take to convert the chunk API into a push model? Which stages benefit most from pipelining?
  2. Blocking vs Non-Blocking – Sort/Aggregate must materialize entire inputs, while Filter/Project/Limit can stream. How does batching affect latency and memory usage of blocking operators?
  3. Join Improvements – Sketch a chunk-based block nested loop join: buffer N outer rows, scan the inner once, emit matched chunks.
  4. Vectorization – DataChunk is intentionally row-based for now. How would you extend it to columnar vectors, and what parts of the operator API would change to exploit SIMD?
