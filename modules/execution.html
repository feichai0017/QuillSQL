QuillSQL Internals

Execution Engine

src/execution/ runs vectorized plans using a Volcano-style iterator model. The new architecture layers chunk-based operators on top of the planner’s VecPlan IR so we can incrementally adopt batch execution without leaking storage details into the planner.

Core Components

ComponentRole
VecPlanPlanner-facing IR that stores metadata (schemas, expressions, table refs) for each operator variant.
ExecPlanImmutable tree of chunk-based operators built from a VecPlan.
Operator traitNew interface (open/get_chunk/rescan/close) that produces DataChunks. Implemented by rewritten operators (SeqScan, IndexScan, Filter, Project, Limit, Sort, Aggregate, DML, and DDL/utility nodes such as Create/Drop Table/Index and Analyze).
OperatorStatePer-execution mutable state owned by an operator (cursor positions, buffers, etc.).
DataChunkRow-oriented batch abstraction (default 1024 rows) exchanged between operators.
ExecutionContextShared context carrying the catalog, TxnRuntime, storage engine, and expression helpers.
TupleStreamStorage-layer scan API (unchanged) used by logical scan operators.

Execution Flow

  1. Planner produces a VecPlan.
  2. ExecPlan::from_vec rewrites every node into its chunk operator counterpart. Each Vec node holds only metadata; all runtime state lives inside operators.
  3. ExecutionEngine::stream_exec_plan opens the root operator, hands it an OperatorState, and returns a ChunkStream iterator. Each call to next() pulls one DataChunk.
  4. ExecutionEngine::execute simply drains the stream into a Vec<Tuple> for callers that still expect materialized results.
  5. ExecutionContext supplies transaction snapshots, lock helpers, and expression evaluation to every operator call.

Operator Examples

  • SeqScanOperator – opens a TableAccess stream, fetches tuples in batches, and hands visible rows downstream without the mutex overhead in the legacy executor.
  • IndexScanOperator – pulls ranges from secondary indexes using IndexScanRequest and applies MVCC visibility checks in batches, honouring table locks for stricter isolation levels.
  • FilterOperator – pulls chunks from its child and evaluates predicate expressions in batch, buffering overflow tuples when the parent chunk is full.
  • ProjectOperator – materializes new tuples by evaluating projection expressions over each chunk.
  • LimitOperator – tracks offset/limit over chunk boundaries before signalling ChunkStatus::Finished.
  • SortOperator – materializes all child chunks once, sorts by OrderByExpr, then streams the sorted tuples in batches (acting as a blocking pipeline boundary).
  • AggregateOperator – consumes all child chunks, updates per-group accumulators, and emits aggregated results chunk-by-chunk (another blocking operator).
  • ValuesOperator / EmptyOperator – source operators that synthesise tuples without reading storage; they now emit batches directly instead of per-row locking.
  • Insert/Update/Delete Operators – run DML in chunk mode, counting affected rows and returning the standard one-row status output without relying on legacy mutex-heavy implementations.
  • DDL & Analyze operators – chunk-friendly wrappers around Create/Drop Table, Create/Drop Index, and Analyze. They execute once, rely on ExecutionContext for catalog and lock helpers, and immediately report ChunkStatus::Finished, keeping transactional semantics identical to the legacy path.
  • NestedLoopJoinOperator – materializes the right input once, then walks left tuples chunk-by-chunk, evaluating the join predicate before emitting merged tuples. The operator keeps Volcano semantics while integrating with chunk scheduling and buffer management.

Interactions

  • StorageEngine – all data access goes through handles/streams, keeping execution storage-agnostic.
  • Transaction – TransactionHandle/TxnRuntime enforce locking, snapshots, and undo logging; operators never talk to LockManager directly.
  • Expression – ExecutionContext::eval_expr / eval_predicate evaluate expressions built by the planner.
  • Optimizer/Planner – execution honours the plan as-is; all structural choices happen upstream.

Teaching Ideas

  • Implement a new chunk operator (e.g., SortOperator or HashAggregate) to exercise OperatorState and chunk-based APIs.
  • Extend ChunkStream to feed HTTP streaming responses without buffering entire result sets.
  • Instrument the operator pipeline (RUST_LOG=execution=trace) to observe chunk flow and rescan behaviour.
